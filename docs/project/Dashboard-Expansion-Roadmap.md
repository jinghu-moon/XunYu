# Dashboard 扩展路线图

这份文档用于回答一个明确问题：**如果要把 `XunYu` 的能力“统统纳入” Dashboard，可视化边界应该怎么划、一级导航应该怎么收、实施顺序应该怎么排。**

当前结论很明确：

- 目标应该是 **“全部能力纳入统一工作台体系”**。
- 不应该走 **“每个 CLI 命令一个一级 Tab”** 的路线。
- Dashboard 更适合做 **本地工作台**，而不是把 CLI 帮助文本逐条翻译成按钮。

## 1. 背景

当前 Dashboard 已经不是一个只做轻量展示的页面集合，而是一个正在成型的本地运维工作台：

- 顶层已有 `Home`、`Bookmarks`、`Ports`、`Proxy`、`Config`、`Env`、`Redirect`、`Audit`、`Diff` 9 个一级面板。
- `EnvPanel` 已经演进成完整的复合工作台，内含变量、PATH、快照、诊断、Schema、Profile、Diff、Audit、注释、模板执行等子能力。
- `DiffPanel` 实际上已经承载了文件浏览、搜索、预览、校验、格式转换和 Diff，不再只是“纯 Diff 面板”。

这说明当前系统已经天然具备两个事实：

1. **CLI 能力并不需要 1:1 对应一个页面。**
2. **工作台式信息架构已经比“命令式面板”更符合项目现状。**

## 2. 目标与边界

### 2.1 目标

- 把适合图形化的 CLI 能力全部纳入 Dashboard 信息架构。
- 保持 `localhost` / `local-first` 的产品边界，不引入远程控制复杂度。
- 复用现有 Axum + Vue 3 的容器型设计，减少重复实现。
- 对危险动作统一加入预览、确认、结果回执和审计链路。

### 2.2 非目标

- 不把所有命令都提升为一级导航。
- 不为了“看起来完整”而做大量低频页面。
- 不在本阶段引入远程认证、多用户权限系统或云同步。
- 不把 `serve` 这类 bootstrap 命令强行做成 Dashboard 内部功能。

## 3. 归并原则

### 3.1 工作台优先，不按命令平铺

一级导航按“工作域”划分，不按 CLI 子命令名铺开。

### 3.2 高频动作前置，低频复杂动作下沉

- 高频：卡片、表格行操作、顶部工具栏。
- 低频：抽屉、对话框、二级页。
- 批处理：单独的批量工作流视图，不打散进多个一级页。

### 3.3 危险动作统一保护

凡是涉及删除、覆盖、批量修改、权限变更、加解密、进程终止的操作，都应统一满足：

- 先预览
- 再确认
- 输出结果摘要
- 可选写入审计流

### 3.4 先复用现有容器模式，再考虑大改路由

当前 `App.vue` 采用单页 + Tab 状态模型，`EnvPanel` / `DiffPanel` 已经证明这种“容器组件 + 子面板”的方案可行。后续优先沿用这套模式，只有当状态规模明显失控时，才考虑切到更重的路由体系。

### 3.5 先接已有后端，再扩新接口

优先消费现有 `/api/*` 能力，再补 CLI 缺失的 Dashboard 专用接口，避免前后端两边同时无序扩张。

## 4. 当前覆盖现状

| 当前一级面板 | 当前已覆盖内容 | 现状判断 |
| --- | --- | --- |
| `Home` | 总览卡片、快捷导航、命令面板 | 适合保留为总览工作台 |
| `Bookmarks` | 书签 CRUD、重命名、批量、导入导出 | 已是“路径与上下文”的核心子域 |
| `Ports` | 端口列表、PID 终止、图标 | 适合并入“网络与代理” |
| `Proxy` | 代理状态、配置、测试、设置/删除 | 适合并入“网络与代理” |
| `Config` | 全局配置读取与 patch | 适合并入“环境与配置” |
| `Env` | 变量治理、快照、Schema、Profile、诊断、审计、Diff、注释、模板执行 | 已经是成熟工作台，应保留为容器结构 |
| `Redirect` | Redirect Profile 管理、dry-run | 适合并入“文件与安全” |
| `Audit` | 审计列表 | 适合升级为跨域“统计与诊断”入口 |
| `Diff` | 文件浏览、搜索、预览、校验、转换、Diff、文件变更监听 | 实际已是“文件工作台”雏形 |

**关键判断：**

- 当前 `Diff` 面板已经溢出“只做 diff”的命名边界，后续应提升为文件相关工作台的承载容器。
- 当前 `Audit` 面板过窄，后续更合理的定位是跨域诊断与审计中心。
- 当前 `Env` 的复杂度说明：以后新增大功能时，应优先复制 `EnvPanel` 这种“一个一级工作台 + 多个清晰子面板”的模式。

## 5. CLI 能力 → Dashboard 工作台总表

| 能力域 | CLI 命令族 | 目标工作台 | 当前状态 | 建议的 Dashboard 形态 |
| --- | --- | --- | --- | --- |
| 书签与路径导航 | `list`、`z`、`open`、`workspace`、`save`、`set`、`delete`、`touch`、`rename`、`tag`、`recent`、`stats`、`dedup`、`export`、`import`、`keys`、`all`、`fuzzy`、`check`、`gc` | 路径与上下文 | 部分已纳入 | 书签列表、工作区切换器、标签看板、最近访问、健康检查、去重/GC、导入导出 |
| 上下文切换 | `ctx` | 路径与上下文 | 未纳入 | Context 列表、切换、重命名、删除、与书签/工作区联动 |
| 代理与网络执行 | `proxy`、`pon`、`poff`、`pst`、`px` | 网络与代理 | 部分已纳入 | 代理状态面板、快速开关、配置模板、检测与测试历史、带代理执行预设 |
| 端口与进程 | `ports`、`kill`、`ps`、`pkill` | 网络与代理 | 部分已纳入 | 端口/PID 表格、筛选、批量终止、进程详情、图标与命令行信息 |
| 全局配置 | `config` | 环境与配置 | 已纳入 | 分段配置编辑器、patch 预览、差异提示、重置与校验 |
| 环境治理 | `env` | 环境与配置 | 已纳入 | 保持现有复合面板结构，继续在二级导航内扩展 |
| 文件浏览与检索 | `tree`、`find`、`bak` | 文件与安全 | 间接已纳入 | 文件浏览器、目录树、搜索结果、备份策略、备份还原入口 |
| 文件改名与删除 | `rm`、`lock`、`mv`、`renfile` | 文件与安全 | 未纳入 | 批量重命名/移动、锁检测、删除预览、冲突提示、结果报告 |
| 文件保护与加密 | `protect`、`encrypt`、`decrypt` | 文件与安全 | 未纳入 | 保护策略卡片、加解密任务表单、目标预览、失败回滚提示 |
| ACL 治理 | `acl` | 文件与安全 | 未纳入 | 查看、增删、继承、Owner、Effective、Diff、Batch、Backup/Restore、Audit、Config |
| Redirect / Diff / 校验 | `redirect`、`diff` | 文件与安全 | 部分已纳入 | Redirect Profile、dry-run、语义/文本 Diff、文件预览、格式转换、配置校验 |
| Shell 集成 | `init`、`completion`、`complete` | 集成与自动化 | 未纳入 | Shell 安装向导、补全脚本生成/复制、补全测试与排错 |
| 别名体系 | `alias` | 集成与自动化 | 未纳入 | Alias 列表、应用扫描、同步、导入导出、which 查询 |
| 批量重命名 | `brn` | 集成与自动化 | 未纳入 | 规则构建器、实时预览、批量执行、回滚提示 |
| 媒体处理 | `img`、`video` | 媒体与转换 | 未纳入 | 图像/视频任务表单、预设、批量队列、输出对比与日志 |
| 内容统计与画像 | `cstat` | 统计与诊断 | 未纳入 | 目录统计、体积分布、内容画像、导出报告 |
| 审计与健康 | `check`、`proxy test`、`env doctor`、当前 `audit` 相关能力 | 统计与诊断 | 部分已纳入 | 跨域健康中心、统一审计时间线、任务日志、失败回放 |
| Dashboard 启动 | `serve` | 不纳入工作台 | 不适用 | 保持 CLI 启动入口，不做 Dashboard 内部页面 |

## 6. 目标一级工作台信息架构

建议把后续 Dashboard 稳定为 **8 个一级工作台**：

| 一级工作台 | 吸收当前面板 | 主要承载能力 | 说明 |
| --- | --- | --- | --- |
| 总览 | `Home` | 全局状态、快捷动作、最近任务、风险提示 | 只做入口与态势，不承载深编辑 |
| 路径与上下文 | `Bookmarks` | 书签、标签、工作区、上下文、导入导出、快速打开 | 路径导航与上下文切换统一收口 |
| 网络与代理 | `Ports`、`Proxy` | 代理、端口、进程、网络测试 | 把“网络开关”和“端口占用”放进一个域 |
| 环境与配置 | `Config`、`Env` | 全局配置、环境变量、PATH、快照、Schema、Profile、诊断 | 保留 `EnvPanel` 的容器型设计思路 |
| 文件与安全 | `Redirect`、`Diff` | 文件浏览、搜索、备份、删除、锁、保护、加解密、ACL、Redirect、Diff、校验 | 这是后续扩展收益最高的工作台 |
| 集成与自动化 | 无现成一级面板 | shell init、completion、alias、批量重命名 | 以“安装与自动化”为主，不再散落在 CLI 语义里 |
| 媒体与转换 | 无现成一级面板 | 图片、视频、格式转换 | 媒体处理与格式转换统一成任务型工作台 |
| 统计与诊断 | `Audit` | 审计、健康检查、运行日志、目录统计 | 做成全局诊断中心，而不是单一审计列表 |

## 7. 哪些能力不建议做成一级 Tab

有些 CLI 命令应当保留为“动作”，而不是做成一级导航：

| 命令 / 能力 | 建议形态 | 原因 |
| --- | --- | --- |
| `serve` | 仅 CLI 启动命令 | Dashboard 无法在自身内部替代自身启动 |
| `open` | 书签表格行操作 / 快捷动作 | 属于路径域中的即时动作，不需要独立页面 |
| `px` | 网络与代理工作台中的执行动作 | 本质是“带代理执行命令”，不是信息域 |
| `kill`、`pkill` | Ports 表格批量操作 | 这是对选中对象的操作，不应独立成页 |
| `touch`、`rename`、`delete`（书签） | 书签行操作 / 批量抽屉 | 属于书签生命周期动作 |
| `completion`、`complete` | Shell 集成工作台中的生成/测试工具 | 适合做向导，而不是一级导航 |
| `pon`、`poff`、`pst` | Proxy 面板中的快捷操作 | 是状态动作，不是页面 |

## 8. 推荐实施顺序

### Phase 0：信息架构重命名与收口

目标：先把“现在已有的能力”放进稳定的工作台框架里，不急着新增大量后端接口。

- `Home` 保留为“总览”
- `Bookmarks` 并入“路径与上下文”
- `Ports` + `Proxy` 并入“网络与代理”
- `Config` + `Env` 并入“环境与配置”
- `Redirect` + `Diff` 并入“文件与安全”
- `Audit` 升级为“统计与诊断”

**交付标准：** 一级导航语义稳定，已有页面都能找到明确归属。

### Phase 1：文件与安全工作台优先扩张

这是最值得优先落地的一期，因为现有 Dashboard 已经具备大量基础 API：

- `/api/files`
- `/api/files/search`
- `/api/info`
- `/api/content`
- `/api/diff`
- `/api/convert`
- `/api/validate`
- `/api/redirect/*`

这一期优先补：

- `tree`
- `find`
- `bak`
- `rm`
- 文件批量操作骨架

**交付标准：** 文件工作流不再被误认为只是 Diff 页面。

### Phase 2：ACL / 锁 / 保护 / 加密纳入文件治理闭环

这一期补齐“文件与安全”真正的治理能力：

- `acl`
- `lock`
- `mv`
- `renfile`
- `protect`
- `encrypt`
- `decrypt`

**交付标准：** 危险动作具备预览、确认、结果回执和审计记录。

### Phase 3：路径与上下文工作台补齐

这一期把书签系统从 CRUD 扩展成真正的“路径中控台”：

- `ctx`
- `workspace`
- `recent`
- `stats`
- `dedup`
- `check`
- `gc`
- `keys` / `all` / `fuzzy`

**交付标准：** 路径、工作区、上下文和标签形成统一操作闭环。

### Phase 4：集成与自动化工作台落地

这一期纳入当前仍偏 CLI-only 的开发辅助能力：

- `init`
- `completion`
- `complete`
- `alias`
- `brn`

**交付标准：** 新用户可通过 Dashboard 完成 shell 集成、补全安装、别名同步和批量重命名。

### Phase 5：媒体与转换 + 统计与诊断完善

最后补齐任务型与分析型能力：

- `img`
- `video`
- `cstat`
- 全局审计时间线
- 任务日志 / 失败日志 / 健康中心

**交付标准：** Dashboard 从“配置与管理界面”升级成完整的本地运维工作台。

## 9. 前后端落地建议

### 9.1 前端

- 保持“一级工作台 + 二级子面板”的结构，不急着把每个能力拆成独立路由。
- 继续复用 `EnvPanel` / `DiffPanel` 这种容器组件模式。
- 抽出统一的危险动作确认组件，避免删除、ACL、加解密、进程终止各写一套。
- 抽出统一任务回执组件，用于展示 `成功 / 部分成功 / 失败 / 回滚提示`。

### 9.2 后端

- 继续按域维护 `src/commands/dashboard/handlers/` 下的模块边界。
- 新增能力时优先按工作域拆分，而不是按零散命令拆分。
- 所有 destructive API 优先支持 dry-run / preview 请求模型。
- 审计流建议统一化，避免 Env、文件、代理各自产生日志但无法归并。

## 10. 完成判定

当以下条件同时满足时，可以认为“统统纳入”的路线图落地正确：

- `src/cli.rs` 中现有顶层命令族，都能在 Dashboard 中找到明确归属，或有充分理由保持 CLI-only。
- 一级导航稳定控制在 **8 个左右**，没有重新膨胀成命令列表。
- 所有高风险操作都具备 **预览 + 确认 + 回执**。
- `Env`、`Diff` 这类复合容器模式被复制到新工作台，而不是让 `App.vue` 继续无限增肥。
- 文档索引、README 和组件理解文档都能解释这套信息架构。

## 11. 当前建议结论

如果只给一句执行建议，就是：

> **先把 Dashboard 从“9 个页面”整理成“8 个工作台”，再按文件与安全 → 路径与上下文 → 集成与自动化 → 媒体与统计的顺序扩张。**

这样既能覆盖全部能力，又不会把 Dashboard 做成难维护的“命令按钮墙”。
