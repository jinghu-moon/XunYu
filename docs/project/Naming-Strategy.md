# XunYu 命名与命令策略

这份文档用于固定项目的品牌名、正式命令名、兼容命令名与快捷别名，避免 README、发布说明、截图、脚本和日常沟通出现混用。

## 当前决策

- 项目名：`XunYu`
- 正式命令名：`xun`
- 可接受备选：`xyu`
- 快捷别名：`xy`

## 当前落地状态

- `xun`：主命令，所有正式文档与示例统一使用它。
- `xyu`：已提供兼容二进制入口，复用同一份 Rust 主入口。
- `xy`：作为高频使用快捷别名，通过 `xun init powershell|bash|zsh` 自动注入。

## 为什么项目名使用 `XunYu`

`XunYu` 比单独的 `xun` 更适合作为仓库和产品名，用于：

- 仓库标题与 README 标题
- 发布页名称
- Dashboard 窗口标题
- 文档与截图标题

它回答的是“项目叫什么”，而不是“终端里敲什么命令”。

## 为什么主命令继续使用 `xun`

`xun` 仍然是当前最合适的正式命令名，因为它：

- 足够短，CLI 输入成本低
- 已和现有命令示例、心智模型、历史使用方式对齐
- 比 `xy` 更有辨识度
- 比 `xyu` 更适合作为主入口

因此当前主命令口径保持为 `xun`。

## `xyu` 的定位

`xyu` 是兼容入口，不替代 `xun` 的正式地位。

适用场景：

- 与品牌名 `XunYu` 对齐的可执行文件名
- 发布包中的兼容命令
- 平滑迁移或额外分发入口

当前仓库已经提供 `xyu` 二进制入口，但所有正式示例仍优先写 `xun`。

## `xy` 的定位

`xy` 只作为快捷方式，不作为主命令宣传。

适用场景：

- 高频使用者减少输入成本
- 已加载 shell 集成后的日常操作

不适合作为主命令的原因：

- 语义弱
- 搜索性差
- 更容易与其它缩写冲突

## 文档与发布规范

- README 标题、项目标题统一使用 `XunYu`
- 所有正式命令示例统一使用 `xun`
- `xyu` 只在兼容命令、发布说明、迁移说明中出现
- `xy` 只在 shell 集成、快捷方式说明中出现

## 推荐表达方式

- 项目名：`XunYu`
- 正式命令：`xun`
- 兼容命令：`xyu`
- 快捷别名：`xy`

## 一句话结论

`XunYu` 负责品牌表达，`xun` 负责正式命令，`xyu` 负责兼容入口，`xy` 负责高频场景下的输入效率。
