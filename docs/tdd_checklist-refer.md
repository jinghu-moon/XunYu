# [模块/功能名称] TDD 开发任务清单

## 🎯 阶段 0：需求与环境对齐 (Alignment)
- [ ] **确认输入**：阅读并理解该模块的开发文档/接口定义。
- [ ] **初始化测试脚手架**：创建测试文件（如 `xxx.spec.ts` 或 `xxx_test.rs`）。
- [ ] **运行基准测试**：确保当前测试套件可以正常运行且没有未解决的错误。

---

## 🔴 阶段 1：编写失败的测试 (Red)
*目标：仅编写足以让测试失败的代码，验证测试的有效性。*

- [ ] **定义测试用例 1：[描述最基础的快乐路径/核心功能]**
  - [ ] 编写前置条件（Arrange）。
  - [ ] 执行目标函数/方法（Act）。
  - [ ] 断言预期结果（Assert）。
- [ ] **运行测试**：观察并确认测试**失败**（编译错误或断言失败）。
  - *预期错误信息：[在此记录预期的失败提示，如“函数未定义”或“返回值不匹配”]*

---

## 🟢 阶段 2：最小化实现 (Green)
*目标：以最快、最简单的方式让测试通过，暂不考虑代码的优雅或极致性能。*

- [ ] **编写实现代码**：在目标文件中编写刚好能让“测试用例 1”通过的逻辑。
- [ ] **运行测试**：观察并确认测试**通过**。
- [ ] **回归检查**：运行全局测试，确保本次修改没有破坏现有功能。

---

## 🔵 阶段 3：重构与优化 (Refactor)
*目标：在有测试保护网的前提下，提升代码质量、性能和可读性。*

- [ ] **消除代码坏味道**：提取重复逻辑，优化变量与函数命名。
- [ ] **架构对齐**：确保代码符合项目整体的架构规范与设计体系。
- [ ] **性能调优**（如适用）：在保证测试通过的前提下，替换为更高效的原生函数或优化内存占用。
- [ ] **运行测试**：确认重构后所有测试依然**100% 通过**。

---

## 🔄 阶段 4：边界与异常处理 (循环 Red-Green-Refactor)
*重复上述红绿重构循环，覆盖非核心路径。*

- [ ] **测试用例 2：[边缘情况 1，如空输入、极大值等]**
  - [ ] [Red] 编写测试并确认失败。
  - [ ] [Green] 完善实现使测试通过。
  - [ ] [Refactor] 梳理异常处理逻辑。
- [ ] **测试用例 3：[异常抛出/错误处理机制]**
  - [ ] [Red] 编写断言捕获错误的测试并确认失败。
  - [ ] [Green] 实现错误抛出逻辑使测试通过。
  - [ ] [Refactor] 统一错误码或错误响应格式。

---

## 🏁 阶段 5：交付与文档同步 (Delivery)
- [ ] **代码清理**：移除所有调试用的打印语句（如 `console.log` / `println!` / `dbg!`）。
- [ ] **文档更新**：如果实现过程中发现原设计文档有遗漏或不合理之处，反向更新开发文档。
- [ ] **提交 Commit**：按照规范提交代码（建议按 TDD 步骤拆分 Commit，例如 `test: add spec for X` -> `feat: implement X` -> `refactor: optimize X`）。
