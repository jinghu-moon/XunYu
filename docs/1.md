## 四、优化方案

### 🔴 P0：高收益优化

#### 1. IO 线程数自动调优

**问题**：`io_threads()` 中 `limit_unc=false` 时 `max=4`（硬编码），但基准显示 8 线程快 20%。

```rust
// parallel.rs:231 — 当前代码
let mut max = if limit_unc { 4 } else { 4 };  // 两个分支一样！
```

**建议**：非 UNC 场景默认用 `min(available_parallelism / 2, 8)`，UNC 保持 4 上限：

```rust
let base = if limit_unc { 4 } else { avail.min(8) };
```

**预期收益**：18ms → 15ms（约 -17%）

#### 2. 消除 dedupe 阶段的 Vec<u16> 分配

**问题**：`normalize_for_dedupe()` 每条路径分配一个新 `Vec<u16>` 作为 IndexSet key（2.5ms / 5000 条）。

**建议**：改用**就地哈希**——在 scratch buffer 中原地 normalize，计算 hash 后与已有 hash 比对，仅在冲突时分配完整 key：

```rust
use std::hash::{Hash, Hasher};
use std::collections::hash_map::DefaultHasher;

fn hash_normalized(units: &[u16]) -> u64 {
    let mut hasher = DefaultHasher::new();
    for &unit in units {
        let unit = if unit == FSLASH { SLASH } else { unit };
        let unit = if (b'A' as u16..=b'Z' as u16).contains(&unit) {
            unit + 32
        } else {
            unit
        };
        unit.hash(&mut hasher);
    }
    hasher.finish()
}
```

用 `IndexSet<u64>` 替代 `IndexSet<Vec<u16>>`，dedupe 分配从 O(n·path_len) 降为 O(n·8)。

**预期收益**：dedupe 阶段 2.5ms → <0.5ms

#### 3. ensure_long_prefix 避免 splice

**问题**：`Vec::splice(0..0, prefix)` 导致整个缓冲区 memmove，O(n)。

**建议**：改为 copy_within + 前置写入：

```rust
fn ensure_long_prefix(wide: &mut Vec<u16>) {
    // ... 已有前缀检测 ...
    let prefix: &[u16] = &[SLASH, SLASH, QMARK, SLASH];
    let old_len = wide.len();
    wide.resize(old_len + prefix.len(), 0);
    wide.copy_within(..old_len, prefix.len());
    wide[..prefix.len()].copy_from_slice(prefix);
}
```

**预期收益**：微秒级，但对 5000+ 路径累积有效

### 🟡 P1：中等收益优化

#### 4. probe_ex 返回冗余结构

`probe_ex` 调用 `GetFileAttributesW` 只获取 `attr`，却构造完整的 `WIN32_FILE_ATTRIBUTE_DATA`（含 6 个零字段）。下游只用 `dwFileAttributes`。

**建议**：直接返回 `u32`，或改为调用 `GetFileAttributesExW` 获取完整数据（如果后续需要时间戳）。当前场景用 `probe()` 即可。

#### 5. 目录级批量探测

**依据**：[C++ Stories 文章](https://www.cppstories.com/2024/cpp-query-file-attribs-faster/) 显示 `FindFirstFileEx(FindExInfoBasic)` 比逐个 `GetFileAttributesW` 快 **50-113x**。

**适用场景**：`acl batch` 或 `delete` 命令输入的路径往往集中在少数目录下。

**实现思路**：
1. 按父目录分组路径
2. 同一目录下路径数 > 阈值（如 10）时，`FindFirstFileEx` 一次遍历整个目录
3. 从遍历结果中匹配目标文件，否则回退到逐个 probe

**预期收益**：集中目录场景下可减少内核调用一个数量级

#### 6. NtQueryAttributesFile 替代 GetFileAttributesW

`GetFileAttributesW` 内部做路径解析 + 安全检查，而 `NtQueryAttributesFile` 是更底层的 ntdll 调用。

由于项目已在 `ensure_long_prefix` 中补充 `\\?\` 前缀（告诉内核跳过路径解析），**这个优化的边际收益较小**。但对 UNC 路径和极长路径仍可能有改善。

**风险**：非文档化 API，Windows 更新可能变更行为。仅建议在 UNC 限流场景下考虑。

#### 7. GetFileInformationByName（Windows 24H2+）

[Rust 标准库 issue #130169](https://github.com/rust-lang/rust/issues/130169) 讨论了这个新 API——无需打开句柄即可获取文件属性，减少 2-3 次系统调用。

**限制**：仅 Windows 11 24H2+ 可用，FAT32 不支持。可作为运行时条件优化路径。

### 🟢 P2：低收益 / 长期优化

#### 8. 并行阈值下调

当前 `PARALLEL_MIN=64`，但 I/O 密集型任务即使 32 条路径也能从并行受益。可考虑降低到 **32**。

#### 9. TLS 缓冲区初始容量

当前 WIDE_BUF 初始容量 512（1 KB），大多数路径 < 260 字符。改为 280 可减少少量内存浪费，但影响极小。

#### 10. 清理死代码

- `to_wide_with_prefix`（`winapi.rs:25`）已无调用方，应删除
- `probe_ex` 是否可简化为 `probe` 直接返回 `u32`
